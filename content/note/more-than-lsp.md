> **TL;DR** — Agent의 stateless한 호출 모델에 LSP의 stateful 세션을 그대로 노출하면 cold-start 및 일관성 비용이 발생한다. clangd-query, Serena, agent-lsp는 이러한 비용을 각자 다른 전략으로 다룬다.

## 1. 들어가며

앞 글에서는 분석형 도구로 분류한 CodeQL과 Glean을 살펴봤다. 둘 다 코드를
데이터로 보고 쿼리하는 모델 — *"코드베이스에 X 패턴이 있는가"* 를 묻는 도구
들이다. 다만 두 도구 모두 도입 부담이 만만치 않았다. CodeQL은 CLI와 쿼리
라이브러리는 오픈소스지만 핵심 엔진이 GitHub의 비공개 자산이고, 상업적 사용
라이선스가 GitHub의 정책에 묶여 있다는 점이 걸렸다. Glean은 Meta 내부에서
거의 단독으로 개발되고 있고, C++ indexer는 별도로 만들어 붙여야 하는 구조
라는 점이 걸렸다. 결국 *세상에 쉽게 주어지는 것은 없다*는 깨달음으로 글을
닫았고, 한 단계 눈높이를 낮춰보기로 했다.

그래서 이번 글에서는 LSP를 넘어 *agent friendly한 LSP* 를 제공하는 도구들을 알아보고자 한다.
LSP 자체는 에디터-언어 서버 간 통신 표준으로 자리잡은 지 오래된 기술이지만, agent 입장에서 LSP를 있는 그대로 쓰기에는 모자란 부분들이 있다.

이번에 다룰 세 도구들 — clangd-query, Serena, agent-lsp — 는 이러한 간극을
메워 LSP의 의미 정보를 agent가 호출 가능한 형태로 재포장한다. 모두 C++를
지원하고 2026년 현재 활발히 개발 중이라는 공통점이 있으나, 간극을 메우는 방식에는 차이가 있다.

먼저 LSP를 agent가 그대로 사용하는 것이 왜 어려운지를 알아보고, 다음으로
이 세 도구들이 그 문제를 어떻게 푸는지 살펴보자.

## 2. LSP와 agent 사이의 간격 — stateful vs. stateless

LSP는 본래 에디터와 언어 서버 사이의 stateful 세션 프로토콜이다. 클라이언트
(에디터)가 `textDocument/didOpen`으로 파일을 열고, 편집하면서 `didChange`로
변경을 흘려보내고, `didClose`로 세션을 닫는다. 서버는 클라이언트가 관리하는
이 파일 상태에 의존해 incremental하게, 즉 변경된 부분만 다시 계산해서 응답한다.
같은 파일을 두 번째로 열면 이미 인덱싱된 상태가 살아 있어 빠르게 답할 수 있다.

반면 agent의 호출은 stateless에 가깝다. 매번
독립적인 도구 호출이 들어오고, 호출 간에 세션 상태가 자연스럽게 유지되지
않는다. 때문에 agent에게 LSP를 있는 그대로 노출하면 두 가지 비용이 발생한다.

첫째는 cold-start 비용이다. 매 호출마다 언어 서버를 새로 띄우고 인덱싱을
처음부터 하면, 대규모 코드베이스에서는 호출 한 번에 수 분이 걸릴 수도 있다.
LSP의 incremental 모델이 주는 이득이 매번 버려지는 셈이다.

둘째는 일관성 비용이다. agent가 LSP의 stateful 프로토콜을 흉내내려면
어댑터 레이어가 필요한데, 그 레이어가 잘못 동작하면 세션 상태가 꼬인다.
파일을 열지 않은 채로 변경 알림만 보낸다거나, 서로 다른 파일 버전을 참조하는
응답이 섞이면 agent는 잘못된 정보 위에서 결정을 내리게 된다.

이 두 비용을 어떻게 다루느냐가 이번 글에서 다룰 세 도구의 설계 결정을
가르는 지점이다. 가장 가벼운 방식은 LSP를 거의 그대로 노출하되 daemon
캐시로 cold-start만 푸는 쪽이고, 무거운 방식은 LSP 위에 자체 runtime을
얹어 세션 상태와 workflow를 도구 차원에서 관리하는 쪽이다.

![LSP의 stateful 세션과 agent의 stateless 호출의 어긋남](/images/lsp-session-gap.svg#center)

각 도구가 이 간격을 어떤 방식으로 메우는지가 다음 장의 주제다.

## 3. 얇은 layer — clangd-query

가장 가벼운 해결책부터 보자. clangd-query는 C++ 전용 CLI 도구로, clangd
위에 얇은 layer를 얹은 형태다. LSP의 기능을 거의 그대로 노출하되, 앞 장에서
짚은 두 비용 중 cold-start 문제를 daemon 캐시를 이용하여 해결한다.

### 동작 방식

clangd-query는 클라이언트/서버 구조로 동작한다. 첫 호출이 들어오면 백그라운드
daemon이 뜨면서 프로젝트의 CMakeLists.txt를 찾아 `compile_commands.json`을
생성하고, 그 위에서 clangd를 띄운다. 두 번째 호출부터는 이미 떠 있는 daemon에
붙어 빠르게 응답한다. 30분 동안 호출이 없으면 daemon이 자동으로 종료된다.

이 구조 덕분에 *"매 호출마다 인덱싱"* 문제는 사라진다. 첫 호출 비용은
발생할 수 밖에 없지만, 이어지는 호출은 warm 상태의 인덱스 위에서 처리되기 때문에 상대적으로
낮은 비용으로 해결할 수 있다. 한 daemon이 일관된 상태를 들고 있으니 일관성 비용 또한 줄일 수 있다.

### 인터페이스

흥미로운 설계 결정 하나는 MCP가 아니라 CLI를 인터페이스로 선택했다는 점이다. 도구 저자는
*"agents seem to have an easier time using command-line tools"* 를 설계 동기로
든다. agent가 도구를 호출할 때 MCP보다 CLI 쪽이 더 자연스럽다는 관찰이다.
주요 명령은 `search`, `show`, `logs` 정도로 단출하고, 출력도 토큰 절약을
위해 의도적으로 짧게 만들어져 있다.

### 한계

C++ 외 언어는 지원하지 않는다. 이름 그대로 clangd 위에서만 동작하기 때문에,
repository에 다른 언어가 섞여 있다면 별도 도구가 필요하다.

CLI 인터페이스는 장점이자 한계이기도 하다. MCP 클라이언트에서 직접 호출
하려면 별도의 wrapping이 필요하고, LSP의 stateful 기능 — 예를 들어 다단계
rename 같은 작업 — 은 노출되지 않는다.

규모도 작다. MIT 라이선스로 활발히 개발되고 있지만, 비교적 작은 프로젝트이고
한 명의 개인 개발자가 주도하는 형태다. 도입 결정 시 이 점을 의식할 필요가
있다.

### 요약

clangd-query는 LSP를 거의 그대로 노출하되 daemon 캐시로 cold-start를 적극적
으로 다루는, *얇은 layer* 에 해당하는 도구다. C++ 전용이고 CLI 기반이라
도입이 가볍지만, 다언어 환경이나 LSP 너머의 기능이 필요한 경우에는 다음 장에서
다룰 두꺼운 layer 도구들로 눈을 돌릴 필요가 있다.

## 4. 두꺼운 layer — Serena, agent-lsp

clangd-query는 cold-start 문제만 풀고 나머지는 LSP에 맡기는 가벼운 길을 택했다.
이번 장에서 다룰 두 도구는 한 발 더 나아가 LSP 위에 자체 runtime과 도구
세트를 얹음으로써, 단순히 LSP를 노출하는 것을 넘어 agent의 작업 흐름 자체를
설계 대상으로 삼는다.

이런 두꺼운 layer가 다루는 영역은 LSP가 본래 제공하던 것 너머에 있다. 여러
호출에 걸친 작업의 세션 영속화, 도구 호출 순서가 의미를 가지는 workflow의
일관성 강제, 변경이 미치는 영향을 미리 가늠해보는 안전망 같은 것들이다.
Serena와 agent-lsp는 같은 *두꺼운 layer* 범주에 속하지만, 이 영역들
중 어디에 무게를 두느냐가 다르다. Serena는 세션 영속화와 다언어 지원에,
agent-lsp는 변경 안전성과 workflow 강제에 무게가 실려 있다.

![clangd 위에 세 도구가 얹는 layer 비교](/images/lsp-tools-layer-stack.svg#center)

### 4.1. Serena

Serena는 스스로를 *"agent를 위한 IDE"* 로 소개한다. 40개가 넘는 언어를
하나의 toolkit으로 묶고, 그 위에 agent가 장시간 작업을 이어갈 수 있도록
세션 영속화와 도구 충돌 관리 layer를 얹은 형태다.

#### 동작 방식

backend는 두 가지를 고를 수 있다. 기본은 무료인 LSP backend로, 언어별로
적절한 언어 서버(C++의 경우 clangd)를 띄워 그 위에서 동작한다. 유료 옵션
으로는 JetBrains backend가 있어, JetBrains IDE의 인덱스를 그대로 활용할 수
있다.

Agent에게 노출하는 도구는 LSP의 raw 호출이 아니라 *심볼 단위* 로 추상화되어
있다. `find_symbol`로 심볼을 찾고, `find_referencing_symbols`로 호출처를
따라가고, `replace_symbol_body`로 본문만 바꾸는 식이다. 파일 단위로 큰
텍스트를 주고받는 대신 심볼 단위로 잘라 다루는 셈인데, 토큰 효율 측면에서
이점이 크다.

#### Memory system

Serena의 특징적인 부분 중 하나는 memory system이다. Agent가 작업 중에 알게
된 사실 — 이 모듈의 구조, 이 함수의 호출 패턴, 진행 중인 작업의 맥락 —
을 별도의 메모리에 적어두고, 다음 세션에서 다시 꺼내올 수 있다. Agent의
컨텍스트는 시간이 지나며 조금씩 사라지지만, 이 memory는 세션을 넘어 계속 유지되는 것이다.

이는 대규모 코드베이스에서 장시간 작업할 때 그 진가가 드러난다. 매번
빈 컨텍스트에서 시작하면 같은 탐색을 반복하게 되는데, Serena가 관리하는 memory가 그 비용을
대신 지불해 주는 것이다.

#### Context mode

Serena는 다른 코딩 agent와 함께 쓰일 수 있다는 점을 의식해서 *context mode*
라는 개념을 둔다. 예를 들어 Claude Code와 함께 쓸 때는 기본 도구(파일 읽기,
편집 등)를 비활성화해서 도구 충돌을 방지한다. Agent 입장에서 같은 일을
하는 도구가 두 벌이면 어느 쪽을 써야 할지 흔들리기 쉬운데, 이 모드가 이런 혼란을
줄이는 데 도움이 될 수 있다.

#### C++ 지원

C++의 경우 clangd 기반으로 동작하고, `compile_commands.json`이 필요하다.
공식 가이드는 셋업에 *5-10분* 정도가 든다고 안내한다. 다른 언어가 보통
바로 동작하는 것에 비하면, C++는 별도의 셋업 단계를 거쳐야 한다는 뜻이다.
빌드 시스템이 CMake가 아닌 환경에서는 별도로 `compile_commands.json` 를 생성해야 한다.

#### 요약

Serena는 *다언어 환경에서 장시간 workflow를 끌고 가는* 데 무게가 실려
있다. C++ 단일 도메인이라면 40개 언어 지원이라는 가치는 거의 활용되지
않지만, memory system과 도구 충돌 관리는 단일 언어 환경에서도 의미가 있다.
이 도구의 강점이 사용자의 환경과 얼마나 겹치는지가 도입 판단의 핵심이 된다.

### 4.2. agent-lsp

agent-lsp는 스스로를 *"가장 완전한 MCP-LSP 서버"* 로 소개한다. Serena가
*agent를 위한 IDE* 를 표방했다면, agent-lsp는 *agent의 안전망* 에 가깝다.
변경이 미치는 영향을 미리 가늠하고, workflow의 순서를 runtime에서 강제하는
쪽에 무게가 실려 있다.

#### 동작 방식

agent-lsp는 단일 Go 바이너리로 배포된다. 내부에 stateful runtime을 두어
언어 서버의 인덱스를 warm 상태로 유지하고, 그 위에 65개 이상의 도구와 24개의
agent workflow를 노출한다. 30개 언어가 CI에서 검증되는데 C++도 그 안에
포함되며, C++ backend로는 clangd를 사용한다.

Agent에게 노출하는 도구도 LSP의 raw 호출이 아니라 구조화된 형태로 추상화
되어 있다. 같은 작업이 grep 기반 탐색에 비해 5-34배 적은 토큰으로 끝난다고
agent-lsp는 주장한다. 이 비율은 코드베이스 크기에 따라 더 벌어질 수 있다.

#### Speculative execution

agent-lsp의 특징적인 기능 중 하나는 바로 speculative execution이다. 디스크에
실제로 쓰기 전에 변경을 미리 *시뮬레이션* 해볼 수 있는 도구를 제공하는 것이다. 단일
변경을 미리 보는 `preview_edit`, 여러 변경이 연쇄적으로 적용됐을 때를 미리
보는 `simulate_chain` 같은 식이다.

이는 Agent가 변경을 시도하기 전에 *이렇게 바꾸면 어떻게 되는가* 를 먼저 묻고
답을 얻을 수 있다는 뜻이다. 실수 비용이 큰 환경 — 운영 중인 코드베이스,
다른 사람의 변경과 섞일 위험이 있는 브랜치 — 에서 그 진가가 드러난다.

#### Phase enforcement

어떤 workflow 에서는 도구 호출 순서가 중요한 의미를 가질 수 있다. 예를 들어 rename을 하기
전에 영향 범위를 확인해야 하고, 변경 후에는 빌드 검증이 따라와야 한다.
만약 agent가 이 순서를 임의로 무시하면 작업 결과가 눈에 띄지 않게 망가질 수 있다.

agent-lsp는 이런 순서를 runtime에서 강제한다. 정해진 phase를 지나지 않은
상태에서는 다음 단계 도구가 호출되지 않거나, 호출이 들어와도 거부된다.
Agent가 자유롭게 활동할 수 있는 범위 *Autonomy* 를 일부 포기하는 대신,
agent가 만들 수 있는 실수의 종류를 줄이는 *Oversight* 설계 결정이라고 볼 수 있겠다.

#### Blast radius 분석

변경이 미치는 영향 범위를 분석하는 도구도 따로 갖추고 있다. 한 심볼을
바꿨을 때 어떤 호출처가 영향을 받는지, 어떤 모듈이 재컴파일되어야 하는지를
agent가 변경 전에 물어볼 수 있다. speculative execution이 *어떻게 바뀌는가*
를 본다면, blast radius는 *어디까지 바뀌는가* 를 본다.

agent-lsp가 자료로 드는 수치 하나가 HashiCorp Consul(약 319,000 라인)에서의
blast-radius 분석 비교다. 같은 분석을 grep 기반으로 수행하면 17.7MB의 출력과
5,534번의 도구 호출이 필요한데, agent-lsp를 거치면 841KB와 119번으로 줄어
든다고 한다. 대규모 코드베이스에서 변경 안전성을 자주 확인해야 하는 환경
이라면 이 차이가 실제 운영 비용으로 누적된다.

#### 요약

agent-lsp는 *변경 안전성과 workflow 강제* 에 무게가 실려 있다. speculative
execution과 phase enforcement는 단순히 *agent를 빠르게 동작시키는* 차원을
넘어, *agent가 만들 수 있는 실수의 종류 자체를 줄이는* 설계다. 변경 비용이
큰 환경, 실수가 조용히 누적되면 곤란한 환경에서 가치가 크다. 다만 가장
최근에 등장한 도구라 운영 사례가 아직 적다는 점은 고려할 필요가 있다.

## 5. 세 도구 비교와 도입 판단

지금까지 세 도구를 차례로 살펴보았다. 같은 *agent friendly한 LSP* 카테고리에
묶이지만 각자가 LSP의 어긋남을 푸는 방식이 다르고, 무게가 실린 영역도
다르다. 이 장에서는 세 도구를 한 자리에 놓고 비교한 뒤, 어떤 환경에서
어느 도구가 더 맞을지 판단 기준을 정리해본다.

### 5.1. 세 도구를 한 표에

| | clangd-query | Serena | agent-lsp |
|---|---|---|---|
| **카테고리** | 얇은 layer | 두꺼운 layer | 두꺼운 layer |
| **초점** | cold-start 해소 | 세션 영속화 + 다언어 | 변경 안전성 + workflow 강제 |
| **인터페이스** | CLI | MCP | MCP |
| **언어 지원** | C++ 전용 | 40+ 언어 | 30+ 언어 (CI 검증) |
| **도구 성숙도** | 개인 개발자 주도, 작은 프로젝트 | 비교적 활발, 유료 backend 옵션 존재 | 가장 최근 등장, 운영 사례 적음 |

표를 살펴보면 같은 카테고리에서도 *초점* 이 다르다는 것이 눈에 띈다. Serena와 agent-lsp
는 같은 *두꺼운 layer* 카테고리에 묶이지만, 추구하는 바는 크게 다르다 — 한쪽은
*세션을 길게 끌고 가는 데*에, 다른 쪽은 *변경 한 번을 안전하게 만드는 데*에
초점을 두고 있다.

*인터페이스* 항목에서는 clangd-query만 CLI를 선택하고 있다. 이 차이는 사소해 보여도
도입 시 의외로 큰 영향을 준다. MCP 클라이언트를 이미 쓰고 있는 환경에서
MCP 도구는 자연스럽게 붙을 수 있는 반면, CLI는 별도 wrapping이 필요하다. 반대로
agent에게 *명령줄 도구* 의 결로 노출하는 것이 더 자연스럽다고 보는 입장도
있고, clangd-query 저자는 그 입장을 명시적으로 택했다.

*도구 성숙도* 는 앞으로 가장 변동이 심할 항목이다. 가장 최근에 나온 도구라는
것은 *기능이 가장 풍부하다* 는 뜻이기도 하고 *운영 중 발견되는 문제가 아직
적게 알려져 있다* 는 뜻이기도 하다. 가장 작은 도구라는 것은 *유지 보수
리스크가 있다* 는 뜻이기도 하고 *코드를 들여다보고 직접 패치하기 쉽다*
는 뜻이기도 하다. 이 항목의 해석은 사용자의 위험 감수 정도에 따라 달라진다.

### 5.2. 어떤 환경에 어느 도구가 맞는가

앞서 살펴본 도구들의 차이점은 일반적인 기준에서의 비교이기 때문에, *어느
환경에서 어느 도구가 맞는가* 에 대한 답을 바로 얻기는 어렵다. 결국 실제로
도구가 사용될 환경을 반영한 기준으로 평가해야 제대로 된 윤곽이 잡힌다.

**언어 구성**

다언어 환경이라면 출발점은 거의 정해진다. clangd-query는 C++ 전용이라
후보에서 빠지고, Serena와 agent-lsp 중에서 선택해야 한다. 단일 언어 환경
이라면 세 도구 모두 선택할 수 있다.

**작업의 길이**

Agent가 짧은 단위 작업을 반복하는 환경(*"이 함수 호출처 보여줘"* 정도)이라면
clangd-query의 가벼움이 잘 맞는다. Agent가 여러 호출을 거치며 한 작업을
끌고 가는 환경 — 모듈을 이해하고, 변경하고, 검증하는 흐름 — 이라면 두꺼운
layer의 가치가 드러난다.

**변경의 비용**

변경이 잘못됐을 때의 비용이 큰 환경 — 운영 중인 코드, 다른 사람의 변경과
섞일 위험이 있는 브랜치, 회귀 발견까지 시간이 걸리는 시스템 — 이라면
agent-lsp의 안전망(speculative execution, phase enforcement, blast radius)
이 잘 맞는다. 변경이 가볍고 되돌리기 쉬운 환경에서는 이런 안전망이 가져다
주는 이점보다 비용이 더 클 수 있다.

**조합 가능성**

세 도구가 꼭 *둘 중 하나* 의 관계인 것은 아니다. 가벼운 탐색은 clangd-query에
맡기고 변경 작업은 agent-lsp로 가는 식의 조합도 가능하다. 다만 도구 충돌
이 생길 수 있으니 Serena의 *context mode* 같은 메커니즘을 의식해서 구성할
필요는 있다.

### 5.3. C++ 단일 도메인이라면

앞서 설명한 기준에 *C++ 단일 도메인* 이라는 조건을 추가하면 선택의 폭은 좀 더 좁아진다.

**`compile_commands.json` 생성 부담**

C++ 환경에서 세 도구 중 어느 것을 고르든 결국 마주하게 되는 허들이 바로
`compile_commands.json` 생성이다. clangd-query는 CMake 프로젝트의 경우 자동
생성을 시도한다는 점에서 가장 부담이 적다. Serena와 agent-lsp는 사용자가 직접
준비하는 것을 가정한다. 빌드 시스템이 CMake가 아닌 환경 — Bazel, MSBuild,
자체 빌드 시스템 등 — 에서는 어떤 도구를 고르든 이 단계가 별도로 필요한 작업이
된다. 도구 선택 이전에 해결해야 할 문제인 셈이다.

**다언어 지원의 활용도**

Serena의 강점인 *40개 언어 지원* 은 단일 언어 환경에서는 거의 활용되지
않는다. 대신 Serena의 다른 강점 — 심볼 단위 추상화, memory system, context
mode — 은 단일 언어 환경에서도 그 자체로 가치가 있다. *다언어가 아니니까
Serena를 빼야 한다* 는 것이 아니라, *Serena를 고르는 이유에서 다언어는
빠진다* 는 의미다.

**clangd 위에 무엇을 더 얹을 것인가**

세 도구 모두 C++ backend로 clangd를 사용하므로, 결국 *clangd 위에 무엇을
더 얹는가* 의 차이가 도구 선택의 본질이 된다. 아무것도 얹지 않은 가장 가까운
형태가 clangd-query, 세션과 memory를 얹은 형태가 Serena, 안전망과 workflow를
얹은 형태가 agent-lsp다. *clangd로 충분한가, 아니면 그 위에 무엇이 필요한가*
가 이 환경에서의 핵심 질문이다.

## 6. 마치며

이번 글에서는 *agent friendly한 LSP* 카테고리에서 세 도구 — clangd-query,
Serena, agent-lsp — 를 살펴보았다. 같은 카테고리에 묶이지만 LSP와 agent
사이의 간극을 메우는 방식과 중점을 둔 목표에서 차이가 있었다. clangd-query는
가장 가벼운 길로 cold-start만 풀었고, Serena는 세션과 memory로 장시간
workflow를 받쳤으며, agent-lsp는 변경 안전성과 phase 강제로 agent의 실수
종류 자체를 줄이려 했다.

앞 글의 분석형 도구(CodeQL, Glean)에 비하면, 이 카테고리는 도입 부담이
확실히 낮은 편이다. clangd라는 잘 알려진 backend 위에서 동작하고, LSP라는
표준을 출발점으로 하기 때문에 *처음부터 만들어야 하는 것* 이 비교적 적다. 다만
*가벼운 도입* 이 *가벼운 가치* 를 뜻하지는 않는다. 두꺼운 layer 도구들이
얹은 세션 영속화, workflow 강제, 변경 안전성은 LSP만 그대로 사용해서는 쉽게 얻을 수 없는 덕목들이다.

세 도구를 두고 *어느 것이 정답인가* 를 묻는 것은 적절한 질문이 아니다.
환경에 따라 — 언어 구성, 작업의 길이, 변경의 비용, 셋업 부담에 따라 — 답이
달라지고, 한 환경에서도 둘을 조합해 쓸 수도 있다. 이 글이 그 판단의
기준을 한 자리에 놓는 데 도움이 됐기를 바란다.
