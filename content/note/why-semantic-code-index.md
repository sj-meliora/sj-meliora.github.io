> **TL;DR.** 대규모 C++ 코드베이스에 agent를 붙이려면 LSP 기반 심볼 단위 질의가 필수적이다. 하지만 LSP는 **심볼 하나를 점찍어 묻는 방식**에 최적화돼 있고, 여러 조건을 조합하거나 범위를 묻는 질문에는 답하지 않는다. 이런 질문들에 답하려면 **코드의 관계를 그래프 DB로 추출해두고 쿼리 언어로 질의하는 모델** — semantic code index — 이 필요하다. 이 글은 왜 그 한 층이 더 필요한지를 설명한다.
>
> | 층위 | 보는 단위 | 답하는 질문 꼴 |
> |---|---|---|
> | 문자열 탐색 | 문자열 | "이 **문자열**이 어디 있는가" |
> | 심볼 단위 질의 | 심볼·AST | "이 **심볼**이 어디서 정의·참조되는가" |
> | 의미론 관계 질의 | 타입·관계·흐름 | "이 **조건들을 모두 만족하는** 코드가 어디 있는가" |

## 1. 왜 이 주제인가

[이전 포스트](https://sj-meliora.github.io/note/context-engineering-for-large-cpp-codebases/)에서 대규모 C++ 코드베이스에 agent를 붙이기 위한 context engineering의 세 축을 정리했다. 그 첫 번째 축이 structural context — LSP를 통해 agent에게 "IDE의 눈"을 달아주는 것 — 였다.

그런데 실제로 LSP 기반 워크플로우를 붙여보면 자연스럽게 올라오는 질문이 있다. **LSP가 주는 정보만으로 정말 충분한가?**

LSP는 "이 심볼 어디서 쓰여?", "이 함수 정의가 어디야?" 같은 질문에는 잘 대답한다. 하지만 실제 작업에서 필요한 질문은 종종 이런 형태이다:

- "이 타입을 반환하는 함수 중에서 특정 락을 잡는 것들"
- "이 설정값이 초기화되는 지점부터 실제로 읽히는 모든 경로"
- "이 인터페이스를 구현하는 모든 클래스 중 특정 패턴을 따르는 것들"

이 질문들에 LSP로 답하려 하면, 결국 여러 번 질의하고 결과를 조합해 추론하는 작업이 된다. 사람이 IDE를 쓸 때는 그럭저럭 감당이 되지만, agent가 같은 일을 할 때는 꽤 비싼 작업이 된다.

이 글은 그 비용이 어디서 오는지, 그리고 왜 LSP 위에 **또 한 층**의 탐색 수단이 필요한지를 설명한다. 구체적인 도구 비교와 도입 판단은 다음 글에서 다룬다.

## 2. 검색 수단의 세 층위

코드베이스를 이해하고자 탐색할 때 우리가 쓰는 도구는 대체로 문자열 탐색, 심볼 단위 질의, 의미론 관계 질의의 세 층위(layer) 중 하나에 속한다. 어느 층위에 있느냐에 따라 답할 수 있는 질문의 종류가 달라진다.

### 문자열 탐색

문자열 탐색에서는 파일을 문자열의 모음으로 본다.

이 층위에 주어지는 질문은 간단하다. "이 문자열이 어디 있는가?" `initialize_dma`라는 문자열을 찾으면 함수 정의, 함수 호출, 주석 안의 언급, 문서 안의 설명이 모두 같은 결과로 돌아온다. 도구는 그게 뭔지 모르고, 알려고 하지도 않는다.

때로는 단순한 것이 최적일 때가 있다. 정규식으로 패턴을 빠르게 긁는 일, 이름을 대충 기억할 때 일단 던져보는 일, 바이너리 파일이나 빌드 산출물까지 포함해 통째로 훑는 일 — 이런 건 단순하게 만들수록 효율적이다.

한계는 두 가지다. 첫째, 동음이의어를 구분 못 한다. `count`라는 변수명을 찾으면 열 개 파일의 스무 개 `count`가 다 나오고, 그 중 어느 게 내가 찾는 그 `count`인지는 사람이 읽어서 골라야 한다. 둘째, 이름이 안 쓰인 관계는 못 찾는다. "이 함수를 호출하는 곳"을 문자열 탐색으로 풀려면 함수 이름을 쿼리로 던져야 하는데, 함수 포인터로 간접 호출되거나 매크로로 감싸여 있으면 놓친다.

### 심볼 단위 질의

심볼 단위 질의에서는 파일을 AST(Abstract Syntax Tree)와 심볼의 모음으로 본다.

이 층위가 답하는 질문은 "이 **심볼**이 어디서 정의되고, 어디서 참조되는가"다. `count`를 찾을 때 어느 `count`인지를 타입과 스코프로 구분한다. 함수 정의로 점프하고, 호출 지점을 모두 나열하고, 타입 계층을 펼치는 동작이 이 층위의 기본 동작이다.

IDE의 "Go to Definition", "Find References", "Rename Symbol"이 전부 이 층위의 기능이다. 이 기능들을 제공하는 대표적 인프라가 LSP — 에디터와 언어 서버 사이의 공통 프로토콜 — 다. 그 외에 tree-sitter 기반의 쿼리 도구들도 같은 층위에 속한다. 구조를 이해하므로 동음이의어에 속지 않는다. AST를 따라가므로 매크로 전개 뒤의 심볼도 (컴파일러가 보는 그대로) 따라간다.

한계는 이 층위가 한 번에 하나의 심볼을 중심에 놓고 묻는 형태 — point query — 에 최적화돼 있다는 점에서 온다. 마치 점묘화처럼, 심볼 하나하나를 점 찍듯이 질의하고 그 결과들을 모아야 비로소 전체 그림이 드러나는 방식이다. "이 위치의 이 심볼에 대해 알려달라"처럼, 심볼 하나를 지정해서 그에 관한 정보를 받는 데에는 최적화되어 있다. 반면 여러 조건을 조합하는 집합 질의나, 값이 흐르는 경로를 따라가는 추적은 이 층위의 사고 방식 자체에 들어 있지 않다. 그리고 이 층위의 도구들은 대체로 **열린 파일 중심의 세션 모델**로 동작한다 — 전체 코드베이스에 대한 관계를 미리 쌓아두고 임의의 질문을 던지는 방식이 아니다.

심볼 단위 질의의 두 한계 — 질문 꼴의 한계, 세션 모델의 한계 — 에서는 뒤에서 좀 더 자세히 다룬다.

### 의미론 관계 질의

의미론 관계 질의에서는 파일을 **타입·관계·흐름**의 모음으로 본다. 더 정확히는, 코드를 관계형/그래프 DB로 추출해두고 그 위에 쿼리를 던지는 모델이다.

이 층위가 답하는 질문은 앞의 두 층위가 못 풀던 조합·흐름 질문이다.

- "`Status` 타입을 반환하면서 내부에서 mutex를 잡는 모든 함수"
- "이 설정값이 초기화되는 지점에서부터 실제로 읽히는 모든 경로"
- "이 가상 함수의 모든 오버라이드 중, 특정 base 메서드를 호출하지 않는 것"

세 질문 모두 여러 조건의 교집합을 구하고, 그래프를 위를 재귀적으로 따라가는 동작을 필요로 한다. 이를 빠르게 푸는 유일한 방법은 코드에서 관계를 먼저 뽑아 DB에 쌓고, 그 DB에 쿼리 언어로 질의하는 것이다.

여기서 용어에 대해 잠깐 짚어두자. 엄밀히 말하면 심볼 단위 질의 층위도 의미론(semantic) 정보를 다룬다 — LSP의 name resolution과 type checking은 컴파일러 의미론 분석의 일부다. 이 글에서 "의미론 관계 질의"라고 부르는 것은 그보다 **한 층 더 깊은 의미론** — 데이터 흐름, 제어 흐름, 함수 경계를 넘는 분석까지 — 을 **관계형 쿼리 형태로** 다룬다는 뜻이다. 그리고 또 하나, 요즘 맥락에서 "semantic search"라고 하면 vector embedding 기반 유사도 검색을 떠올리기 쉬운데, 그 "semantic"과는 다르다. 이 글의 "의미론"은 프로그램 의미론 — 컴파일러가 이해하는 종류의 의미 — 이다. Embedding은 "의미가 비슷해 보이는 것"을 찾고, 이 층위는 "의미론적 관계가 실제로 성립하는 것"을 찾는다. 서로 다른 종류의 도구들이다.

### 정리

| 층위 | 보는 단위 | 답하는 질문 꼴 |
|---|---|---|
| 문자열 탐색 | 문자열 | "이 **문자열**이 어디 있는가" |
| 심볼 단위 질의 | 심볼·AST | "이 **심볼**이 어디서 정의·참조되는가" |
| 의미론 관계 질의 | 타입·관계·흐름 | "이 **조건들을 모두 만족하는** 코드가 어디 있는가" |

각 층위는 이전 층위를 대체하지 않는다. 하위 층위가 여전히 유용하고, 그 위에 다른 질문 꼴이 더 필요할 때 다음 층위가 등장한다.

## 3. 심볼 단위 질의가 하지 않는 것

여기서는 앞서 언급한 심볼 단위 질의의 두 한계 — 세션 모델의 한계, 질문 꼴의 한계 — 가 실제 작업에서 어떻게 드러나는지 알아본다. 먼저 사람이 IDE로 코드를 읽을 때의 관점에서, 그다음 agent가 작업을 계획할 때의 관점에서 생각해 보자. 이하의 논의는 LSP를 대표 예시로 들지만, 같은 한계가 이 층위의 다른 도구들에도 해당된다.

### 세션 모델의 한계: Cross-TU 관계의 영속화가 없다

LSP 서버는 근본적으로 **지금 열려 있는 것** 중심으로 동작한다. 편집 중인 파일이 있고, 그 파일이 include하는 헤더들이 있고, 서버는 그걸 빠르게 해석해서 정의·참조·타입 정보를 돌려준다. 프로젝트 전체를 한 번에 다 인덱싱해두는 서버도 있지만, 그 인덱스조차 **서버 세션의 부속품**이다. 서버가 꺼지면 사라지고, 서버가 켜지면 다시 쌓인다. 

이게 Translation Unit, TU 경계를 넘는 질문에서 문제가 된다. `foo.cpp`에서 정의된 심볼이 `bar.cpp`에서 참조된다고 할 때, 둘 다 같은 컴파일 DB에 들어 있으면 LSP는 이 관계를 안다. 하지만 이 관계를 아는 **상태**는 서버 메모리에만 있다. 외부에서 "이 관계 그래프 전체를 쿼리하고 싶다"고 하면, LSP 프로토콜이 제공하는 인터페이스를 통해 "이 위치의 이 심볼에 대해 알려달라"는 point query를 여러 번 요청한 후 그 결과값을 바탕으로 관계를 구성해야 한다. 관계 그래프 자체를 얻어올 수 있는 방법이 없다.

### 질문 꼴의 한계 (1): 임의의 관계 쿼리

이제 질문 꼴의 한계에 대한 구체적인 예를 살펴보자.

> "이 타입을 반환하는 모든 함수 중에서 내부에 락을 잡는 것"

이 질문의 해는 두 가지 조건의 교집합이다. 조건 A는 "반환 타입이 X", 조건 B는 "본문 어딘가에 `lock` 호출". LSP는 이 질문에 대해 답할 수 있을까? 먼저 A에 대해서는, LSP에게 이를 직접 물을 방법이 없다. "Find References"는 타입의 모든 등장 지점을 주지만 그건 반환 타입인지, 파라미터 타입인지, 로컬 변수 타입인지 구분하지 않는다. B는 아예 LSP의 일이 아니다 — 함수 본문을 내가 직접 읽고 락 호출 여부를 판정해야 한다.

그래서 실제로는 이렇게 된다. LSP로 타입 X의 참조를 다 뽑고, 사람이 눈으로 "이건 반환 타입"인 것만 거르고, 그 함수들을 하나씩 열어 본문에서 락 호출을 찾는다. 사람이 LSP와 grep 사이를 오가며 **쿼리 최적화기** 역할을 한다. 열 개 함수면 할 만하지만, 이 질문이 실제로 값질 때는 보통 수백 개 후보가 있다.

### 질문 꼴의 한계 (2): 데이터 흐름

또 다른 예를 보자.

> "이 설정값이 초기화되는 지점에서부터 실제로 읽히는 모든 경로"

이건 **값의 흐름**을 따라가는 질문이다. 어떤 전역 구조체의 필드가 부팅 시점 어딘가에서 쓰여지고, 그게 나중에 어디서 읽히는지. LSP는 필드의 모든 읽기·쓰기 위치를 돌려줄 수 있지만 — 참조 목록이니까 — **"이 쓰기가 이 읽기로 실제로 흘러가는가"**는 답하지 않는다. 두 이벤트 사이에 덮어쓰기가 있었다면 원래 값은 소멸됐다. 조건 분기에 따라 어떤 경로에서는 쓰이고 어떤 경로에서는 안 쓰인다. 이걸 보려면 제어 흐름과 데이터 흐름을 따라가는 분석이 필요한데, 심볼 단위 질의는 이를 지원하지 못한다.

### 질문 꼴의 한계 (3): 도달 가능성(Reachability) 파악

세 번째 예는 **"이 함수에서 시작해서 도달 가능한 모든 경로"**를 찾는 그래프 탐색 문제다.

> "특정 ISR(Interrupt Service Routine) 핸들러에서 시작해, 최종적으로 blocking 함수(msleep, lock 등)를 호출하게 되는 모든 실행 경로"

스토리지 F/W 맥락에서 이러한 질문은 전형적이다. 만약 하위 함수 중 하나가 blocking 요소를 포함하고 있다면, 그 함수가 ISR 컨텍스트 내의 어떤 경로를 통해 트리거될 수 있는지 전부 파악해야만 잠재적인 하드락(Hard-lock)을 막을 수 있기 때문이다.

LSP는 "이 함수를 호출하는 곳"과 "이 함수가 직접 호출하는 함수들"은 준다. 한 단계 위, 한 단계 아래. 그 이상을 원하면 재귀적으로 파고들어야 하는데, 이건 LSP 프로토콜의 자연스러운 요청 꼴이 아니다. 클라이언트가 직접 BFS를 돌려야 한다 — 매 단계마다 서버에 물어보면서. 함수 포인터나 가상 함수가 끼면 한 호출 지점이 여러 타겟으로 퍼지고, 그걸 정확히 처리하는 건 심볼 단위 질의 바깥의 일이다.

이 세 질문의 공통점은 뭘까? 세 질문 모두, 심볼 단위 질의 층위에서 풀려면 **점을 찍어가며 그림을 그려내듯** 수많은 point query를 엮어 사용자 쪽에서 관계 그래프를 재구성해야 한다. 그리고 그 재구성 과정이 복잡해질수록 클라이언트 쪽 부담이 기하급수적으로 늘어난다. 모두 **필요한 정보를 그래프 형태로 먼저 구성해 두고, 쿼리 언어로 질의하면 좀 더 자연스럽게 풀렸을** 문제들이다.

### Agent 관점: 영향 범위 파악

지금까지는 사람이 IDE로 코드를 읽을 때의 얘기였다. Agent 관점에서는 문제가 한 겹 더 얹힌다.

Agent가 "이 인터페이스의 시그니처를 바꿔 달라" 같은 변경 작업 요청을 받았다고 해보자. 똑똑한 agent가 가장 먼저 해야 하는 일은 **영향 범위 파악**이다. 이 인터페이스를 누가 쓰는가, 어디를 같이 고쳐야 하는가, 테스트는 어디 있나.

이 과정에서 agent가 심볼 단위 질의에 던져야 할 질문은 보통 이런 꼴이다:

- "이 인터페이스를 구현하는 모든 클래스"
- "그 중 이 메서드를 override하면서 특정 패턴으로 쓰는 것들"
- "호출 경로 상에서 이 값이 흘러가는 테스트 케이스들"

앞에서 본 그대로, 이 층위는 이 질문들에 point query의 조합으로만 답한다. 그래서 agent는 LSP에 여러 번 물어보고, 결과를 자기 context에 쌓고, 그 위에서 다시 추론한다. 이 과정이 두 가지 비용을 만든다.

첫째는 **턴 비용**이다. LSP에 N번 질의하고 매번 결과를 받아 해석하면, agent의 컨텍스트가 LSP 응답으로 차오른다. 쿼리를 한 번 잘 던지면 답이 오는 경우와, 스무 번 던져서 조합해야 하는 경우의 컨텍스트 비용 차이가 크다.

둘째는 **공유의 비효율성**이다. 여기서 '공유'란, 여러 사람으로 구성된 조직 단위에서의 공유를 뜻한다. 개인 단위의 컨텍스트 연속성은 비교적 쉽게 유지할 수 있다 — LLM 서비스가 제공하는 memory 기능 활용하기, 세션 간 핸드오프 문서 남기기 같은 방법을 통해 내가 오늘 세운 관계 지도는 내 다음 세션의 컨텍스트로 어떻게든 전달할 수 있다. 하지만 이런 방식을 조직 단위에서 적용하기는 어렵다. 타인에게 컨텍스트를 공유하기 위해서는 여러 가지 번거로운 단계를 거쳐야 할 테고, 그렇게 공유한 컨텍스트 — 내 목적에 맞는 질문으로 구축한 — 가 다른 사람의 작업에 유의미하게 도움이 될 것이라고 보기도 어렵다. "질문" 에 대한 "답"은, 같은 질문을 한 사람에게만 유용한 법이다.

### 정리

이 층위가 잘못 설계됐다는 얘기가 아니다. LSP를 비롯한 심볼 단위 질의 도구들은 **사람이 IDE에서 한 파일을 편집하고자 하는** 요구사항을 만족시키기 위해 만들어졌고, 주어진 요구사항을 잘 만족시킨다. 단지 Agent가 등장하면서 새로운 요구사항이 주어쟜을 뿐이다:

- 코드베이스 전체에 대한 임의의 관계 쿼리
- Agent가 변경 전 영향 범위를 계획하기, 그리고 그 결과를 조직이 공유하기

이 두 요구사항은 필연적으로 "필요한 정보를 그래프로 먼저 구성하고 쿼리 언어로 질의하는" 모델을 요구한다. 그 모델이 바로 semantic code index 이다.

## 4. Semantic Code Index

앞 장의 끝에서 도달한 지점을 다시 한 번 짚자. 심볼 단위 질의의 point query로는 여러 조건의 교집합, 그래프 순회, 데이터 흐름 같은 질문이 풀리지 않는다. 그리고 개인이 LSP 위에서 세운 관계 이해는 세션이 끝나면 사라지고, 조직이 공유하기도 어렵다. 두 문제를 한꺼번에 푸는 방향이 semantic code index다.

이름에 대해 먼저 짚어두자. 업계에서 "semantic code index" 또는 "semantic code search"라는 이름이 굳어져 쓰인다. 엄밀히는 LSP도 의미론(semantic)을 다루므로 이 이름은 오해의 여지가 있지만, 널리 쓰이는 표현이니 이 글에서도 그대로 쓴다. 앞 장들에서 "의미론 관계 질의"라고 부른 것이 이 이름의 실제 내용이다.

### 기본 구조

Semantic code index의 구성 요소는 세 층으로 나뉜다.

1. **추출기 (extractor)**: 코드에서 관계를 뽑아낸다. 컴파일러의 프런트엔드와 비슷하거나 실제로 그 일부다. 파싱·이름 해결·타입 체킹을 거친 결과에서, 심볼·타입·참조·호출·상속·흐름 같은 관계를 구조화된 사실(fact)로 추출한다.
2. **저장소 (store)**: 추출된 사실을 보관한다. 관계형 DB, 그래프 DB, 혹은 이 용도로 설계된 전용 저장소.
3. **쿼리 언어 (query language)**: 저장소 위에서 질문을 던지는 수단. SQL, Datalog, 또는 그 시스템 전용 DSL.

### 심볼 단위 질의와 무엇이 다른가

이 구조가 심볼 단위 질의와 다른 점은 세 가지인데, 세 가지가 독립된 장점이 아니라 **한 가지 전환에서 파생된 연쇄적 결과**다.

그 전환은 이렇다. 심볼 단위 질의의 LSP 서버가 들고 있는 것은 관계 그래프가 아니다. 심볼의 카탈로그, 그리고 각 심볼에 대한 지역적인 이웃 정보(정의 위치, 직접 참조자, 한 단계 위/아래 호출자)다. 사용자는 **점묘화 화가처럼** 필요할 때마다 점을 하나씩 찍어가며 자기 쪽에서 그림 — 관계 그래프 — 을 그려내야 한다. 그리고 그렇게 그려낸 그림은 사용자의 세션 — 개발자의 머릿속, agent의 컨텍스트 — 안에 **휘발적(volatile)으로** 존재한다. 세션이 끝나면 사라진다.

Semantic code index의 출발점은 발상의 전환이다. **매번 점을 찍어가며 그림을 그리는 대신, 그림을 미리 그려둔다.** 코드의 관계를 사전에 그래프로 추출해서 비휘발성(persistent) 저장소 — 별도의 DB — 에 쌓아둔다. 질의 시점에는 이미 그려진 그래프에 선언적 쿼리를 던진다. 이 전환 하나에서 세 가지가 따라온다.

![심볼 단위 질의 vs Semantic Code Index: 점을 찍어가며 그리기 vs 미리 그려둔 그림](/images/semantic-index-diagram.svg)

첫째, **관계 그래프가 영속한다.** 사용자가 세션 안에서 힘들게 재구성하던 것이 DB에 이미 있는 상태다. 같은 관계를 매번 다시 구성할 필요가 없다.

둘째, **쿼리 꼴이 선언적이 된다.** DB가 있으니 그 위에 쿼리 언어를 얹을 수 있다. "이 조건을 만족하는 것들"이라 쓰면 어떻게 찾을지는 시스템이 푼다. 사용자가 LSP와 grep 사이를 오가며 쿼리 최적화기 역할을 할 필요가 없다.

셋째, **결과가 조직이 공유하는 자원이 된다.** 이건 첫째 특성의 자연스러운 귀결이다. 비휘발성 저장소가 있으면 그 DB를 여러 사람이 질의하는 것이 가능해진다. 앞 장에서 짚었던 "공유의 비효율성" 문제 — 개인이 점을 찍어 그려낸 그림이 타인에게 전달되기 어려운 문제 — 가 구조적으로 달라지는 지점이 여기다. 각자의 목적으로 같은 DB에 질의하면 되므로, 애초에 각자가 그림을 다시 그릴 필요 자체가 없어진다.

### 무엇이 가능해지는가

이 구조 위에서 앞 장에서 어려웠던 질문들이 한 번의 쿼리로 풀리기 시작한다.

"Status 타입을 반환하면서 내부에서 mutex를 잡는 모든 함수"는 두 조건의 교집합을 DB에 질의하는 꼴이 된다. 사람이 중간에 개입해 후보를 걸러낼 필요가 없다. "이 설정값이 초기화되는 지점부터 실제로 읽히는 모든 경로"는 데이터 흐름 분석을 1급 쿼리로 지원하는 시스템이라면 그래프 순회 한 번으로 답이 나온다. Call graph 전체 순회도 재귀 쿼리로 자연스럽게 표현된다.

Agent 관점에서의 효과는 더 크다. LSP에 스무 번 질의해서 조합해야 했던 작업이 한 번의 선언적 쿼리로 끝나면, 컨텍스트 소모가 줄고 추론 품질이 올라간다. 그리고 쿼리의 결과가 조직이 공유하는 DB에서 나오므로, 한 사람의 agent 세션이 던진 질문의 결과를 다른 사람의 agent가 곧바로 참조할 수 있다.

### 대표 시스템 셋

이 모델을 실제로 구현한 시스템이 여럿 있다. 특히 세 시스템이 자주 언급된다.

- **CodeQL** (GitHub): 보안 취약점을 코드 패턴으로 정의하고 탐지하는 것이 원점. Datalog 계열 쿼리 언어(QL)를 쓰며, 데이터 흐름 분석을 일급 시민으로 지원한다.
- **Glean** (Meta): 초대형 모노레포에서 "Go to Definition" 같은 IDE 기능을 빠르게 돌리는 것이 원점. 증분·병렬 인덱싱에 강하다.
- **Kythe** (Google): 서로 다른 언어가 섞인 프로젝트에서 언어 경계를 넘는 관계를 하나의 그래프로 다루는 것이 원점. 스키마 표준화에 힘을 쏟았다.

세 시스템은 서로의 대체재가 아니다. 각자가 원래 풀려던 문제가 다르고, 그 차이가 시스템의 성격과 제약을 동시에 만들었다. 그래서 "무엇을 고를 것인가"는 "semantic code index가 필요한가"와는 별개의 질문이 된다.
