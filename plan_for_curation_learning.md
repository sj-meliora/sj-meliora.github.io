# Curation 파이프라인 학습 Plan

## Context

sj-meliora 블로그의 다음 마일스톤은 일일 큐레이션 파이프라인 구현이다. CLAUDE.md 로드맵의 단계 3~4. 이전 세션에서 바로 pipeline 설계 plan으로 들어갔으나, 운영자가 RSS 개념부터 익숙하지 않다고 판단해 **설계 이전에 학습 단계를 먼저 두기로 합의**했다.

이 plan은 구현용이 아니라 학습용이다. 5개 모듈로 끊어 핵심 개념만 빠르게 훑고, 학습이 끝나면 다시 plan mode로 들어와 pipeline 설계에 착수한다.

**운영자 선호사항 (사전 확인됨)**
- 깊이: 각 주제당 1~2시간 분량, 핵심 개념 위주 — 구현 기초는 확보하되 원리까지 파고들진 않음
- 방식: 읽기 + Claude의 설명 중심. Claude가 자료를 선별·번역·요약하고 이해 점검 질문을 던짐. 손으로 하는 실습은 최소
- 자료 언어: 무관. 영어 자료는 Claude가 핵심만 한국어로 번역·요약

---

## 학습 모듈 (총 5개)

### Module 1 — RSS 기초

**학습 목표**
- RSS가 무엇이고 왜 만들어졌는가
- 어떤 사이트가 RSS를 제공하는가, 어떻게 피드를 찾는가
- RSS 문서의 기본 구조 (channel, item, pubDate, link, description)
- RSS 2.0 vs Atom의 차이점 — 개념 수준에서

**전달 방식**
- Claude가 개념 요약 (5~10분 읽는 분량)
- 실제 RSS 피드 URL 2~3개를 같이 열어보며 구조 해설 (BBC, Hacker News, GitHub repo feed 같은 공개 피드)
- 이해 점검 질문 3개

**성공 기준**: 임의의 RSS 피드 URL을 받았을 때, "이게 뭔지, 어떤 정보가 들어있는지, 어떻게 기계가 읽는지" 설명 가능

### Module 2 — 큐레이션을 위한 수집 방법론

**학습 목표**
- RSS 외의 수집 옵션: 웹 스크래핑, REST API, JSON feed, 뉴스레터 파싱, 기존 큐레이션 서비스 (Feedly/Inoreader 등)
- 각 방법의 trade-off: 합법성/ToS, 유지보수 부담, 커버리지, 안정성
- sj-meliora 맥락에서의 선택 기준: "운영자 혼자 유지보수 가능한가"가 최우선

**전달 방식**
- Claude가 옵션별 비교표 제시
- 각 방법에 대해 "어떤 사이트에서 이게 현실적으로 되는가" 구체 예시
- 이해 점검 질문 3개

**성공 기준**: 운영자가 구독하고 싶은 임의의 정보원에 대해 "RSS로 될까, 스크래핑이 필요할까, 아예 불가능한가"를 판별 가능

### Module 3 — LLM으로 가공하는 방법론

**학습 목표**
- Anthropic Messages API의 기본 구조 (system / user / assistant 메시지)
- 프롬프트 설계의 기본 원칙 — 특히 구조화된 출력을 안정적으로 얻는 법
- 톤 캡처 전략 두 축 비교: style guide 명문화 vs few-shot 예시
- 비용 구조 기초: 입력 토큰 vs 출력 토큰, 모델별 단가

**전달 방식**
- Claude가 Anthropic Docs의 핵심만 한국어 요약
- 실제 프롬프트 예시 하나를 같이 읽으며 "왜 이렇게 썼는지" 해설
- 이해 점검 질문 3개

**성공 기준**: 운영자가 "내 톤으로 이 RSS 아이템을 요약해줘"라는 프롬프트를 직접 초안 잡아볼 수 있음

### Module 4 — GitHub Actions로 헤드리스 자동화

**학습 목표**
- 워크플로우·잡·스텝·트리거의 개념 (특히 cron)
- secrets 저장·사용법
- 상태 관리 패턴 (artifact vs repo commit vs 외부 저장소) — 중복 방지를 위해
- 실패 처리와 알림

**전달 방식**
- Claude가 GitHub Docs 핵심 요약
- 저장소의 기존 `.github/workflows/deploy.yml`을 같이 읽으며 문법 해설
- 이해 점검 질문 3개

**성공 기준**: 운영자가 "매일 아침 7시 KST에 스크립트 실행 + 실패 시 이슈 생성"이라는 요구를 받았을 때 워크플로우 구조를 그릴 수 있음

### Module 5 — 통합 관점 복습 + 설계 단계로 전환 준비

**학습 목표**
- 앞 4개 모듈이 하나의 파이프라인에서 어떻게 맞물리는지 전체 흐름 재조립
- 이전 세션에서 Claude가 제시한 설계 D1~D9 (소스 선정, 글 단위, 톤 전략, 실행 시각, 중복 방지, 리뷰 게이트, 실패 처리, 코드 구조, front matter 스키마)를 **지금의 이해 수준으로 다시 읽어본다**
- 이해되지 않거나 결정 근거가 부족한 지점 식별

**전달 방식**
- Claude가 전체 파이프라인 다이어그램 제시
- D1~D9를 하나씩 훑으며 "이제 이 결정을 본인이 판단 가능한가?" 자가 점검
- 미해결 지점을 정리한 상태로 다음 `/model opusplan` 세션(설계 plan)으로 넘어감

**성공 기준**: D1~D9 중 어느 것도 "무슨 말인지 모르겠다"는 지점이 없음. 남은 건 순수한 선택의 문제

## 진행 리듬

- **모듈 단위**: 한 번에 한 모듈씩. 운영자가 "다음 모듈" 신호를 주면 진행
- **세션 단위**: 한 모듈을 여러 세션에 쪼개도 무방. 긴 자료를 읽고 며칠 뒤 돌아오는 경우 Claude가 직전 맥락을 다시 요약
- **질문**: 각 모듈 끝 "이해 점검 질문"은 강제가 아니다. 운영자가 답하고 싶을 때만 답함. 질문은 노출되지 않은 오개념을 걸러내는 용도

---

## 산출물

**학습 중**
- 이 plan 파일이 진행 상태의 ground truth. 각 모듈 완료 시 상태 업데이트는 별도 파일 아닌 이 문서에서 관리

**학습 완료 후**
- Module 5가 끝나면 이 plan mode에서 나가고, 바로 **다음 plan mode 세션**을 `/model opusplan`으로 시작해 pipeline 설계 plan을 만든다 (이전 세션에서 제시했던 D1~D9가 기반)
- 그 설계 plan은 CLAUDE.md convention에 따라 `plan_for_curation_pipeline.md`로 저장됨

## 현재 plan의 저장

이 문서(`starry-finding-horizon.md`)는 plan mode 내부 작업 파일이다. Plan mode 종료 후 sonnet 실행 단계에서 CLAUDE.md convention에 따라 저장소 루트에 **`plan_for_curation_learning.md`**로 복사·커밋한다. 두 파일의 내용은 동일하게 유지.

## 검증

학습 plan의 성공은 이렇게 측정한다.

1. Module 1~4 각각의 "성공 기준"을 운영자 본인이 "충분히 충족했다"고 판단하는가
2. Module 5 끝에 D1~D9 중 이해 불가 지점이 0인가
3. 다음 plan mode 세션에서 pipeline 설계를 진행할 때, Claude의 제안을 운영자가 능동적으로 판단·수정할 수 있는가 (이전 세션처럼 전제 자체가 무너지지 않는가)

이 세 조건이 충족되면 학습 plan은 성공.
