# CLAUDE.md

> 이 저장소에서 작업하는 Claude Code를 위한 컨텍스트 파일.
> 이 문서는 프로젝트의 성격, 아키텍처, 그리고 어떤 수정·기능 작업에서도 지켜져야 할 운영 원칙을 기술한다.

---

## 프로젝트 개요

**sj-meliora**는 개인 큐레이션 블로그다. 1차 독자는 운영자 본인이며, 외부 공개는 부산물이다.

블로그는 단일 주제에 한정되지 않는다. 국제 정세, 경제, 기술 등 여러 도메인을 다룬다. 콘텐츠는 두 갈래의 하이브리드 파이프라인으로 생산된다.

1. **큐레이션 파이프라인 (자동)**: 외부 서비스와 RSS 피드가 매일 원시 정보를 수집한다. LLM이 이를 운영자의 톤으로 가공해 구조화된 초안을 만든다.
2. **노트 파이프라인 (수동)**: 운영자가 비정기적으로 에세이·관찰·생각을 직접 쓴다.

블로그 이름은 라틴어 *Ad Meliora* — "더 나은 것을 향하여" — 에서 가져왔다.

## 기술 스택

| 레이어 | 선택 | 비고 |
|---|---|---|
| 정적 사이트 생성기 | Hugo | 단일 바이너리, 빠른 빌드, Ruby/Node 의존성 없음 |
| 테마 | PaperMod | 미니멀, 본문 중심, 다크/라이트 토글, 사이드바 없음 |
| 호스팅 | GitHub Pages | 무료, 저장소와 통합, 커스텀 도메인 가능 |
| 빌드·배포 | GitHub Actions | `main` 브랜치 push 시 Hugo 빌드 워크플로우 실행 |
| 일일 큐레이션용 LLM | Anthropic API (직접 HTTP 호출) | GitHub Actions에서 호출. 인터랙티브 Claude Code 사용과 분리 |
| 정보 수집 | 외부 서비스 + RSS | 저장소 자체에 스크래핑 로직을 두지 않음 |

### Hugo + PaperMod를 선택한 이유

- **Hugo vs Jekyll**: 빌드 속도가 빠르다(일일 큐레이션 파이프라인은 시간이 지날수록 수백 편의 글을 누적시킬 것이다). 단일 바이너리 설치, Ruby 툴체인 불필요.
- **Hugo vs Next.js / 커스텀 SSG**: 운영자는 유지보수 부담을 0에 가깝게 만들기 위해 의도적으로 "고정된 테마를 가진 정적 사이트 생성기"를 선택했다. 이전에 커스텀 Next.js 구성으로 시도한 적이 있으며, 그 방향은 다시 채택하지 않는다.
- **PaperMod vs Chirpy 등 다른 테마**: 사이드바가 없는 본문 중심 레이아웃. 운영자는 카테고리 트리 내비게이션보다 잡지 형식의 읽기 경험을 선호한다.

### 일일 파이프라인에 Claude Code가 아닌 Anthropic API를 쓰는 이유

- 일일 파이프라인은 GitHub Actions cron으로 헤드리스 실행된다. Claude Code는 인터랙티브 터미널 사용을 전제로 설계되었으며, 헤드리스 실행은 인증·한도·약관 측면의 복잡성을 일으킨다.
- 직접 API 호출은 요청 단위 청구가 예측 가능하고, 운영자의 인터랙티브 Claude Code 사용량과 깔끔하게 분리된다.
- Claude Code는 파이프라인을 *개발*하는 단계와 글을 수동으로 작성하는 단계에서 계속 사용한다.

## 콘텐츠 모델

### 카테고리 — 정확히 2개. 확장 금지.

| 카테고리 | 목적 | 출처 | 빈도 |
|---|---|---|---|
| `curation` | 외부 정보를 운영자의 톤으로 정리한 글 | 자동 파이프라인 + 수동 큐레이션 | 일 단위 (목표) |
| `note` | 운영자의 생각·에세이·관찰 | 수동 | 비정기 |

**카테고리 수는 2개로 고정한다.** 주제별 세분화는 전적으로 태그로 처리한다. `study`, `tech`, `economy`, `personal` 같은 추가 카테고리를 제안하지 말 것 — 이런 것들은 모두 태그의 영역이다.

### 태그 — 자유롭게, 다중 사용

태그가 주제 세분화를 담당한다. 예: `economy`, `AI`, `semiconductor`, `US`, `geopolitics`, `firmware` 등. 한 글에 여러 태그를 달 수 있다. 태그 분류 체계는 자연스럽게 성장하도록 둔다.

### Front matter 규약

모든 글은 최소한 다음 항목을 포함한다.

```yaml
---
title: "글 제목"
date: 2026-04-12T08:00:00+09:00
categories: ["curation"]    # 또는 ["note"]
tags: ["AI", "semiconductor"]
draft: false
---
```

`date` 필드는 **글을 생성하는 시점의 현재 한국 시간(KST, `+09:00`)**으로 채운다. 과거·미래 날짜를 임의로 넣지 않는다. Claude Code가 front matter를 생성할 때는 다음 명령으로 실제 현재 KST를 조회해 반영한다 (Windows/git-bash 환경에서 `TZ=Asia/Seoul date`는 TZ 변수를 제대로 적용하지 못해 UTC를 반환하므로 PowerShell 경유가 안전하다):

```bash
powershell -c "[System.TimeZoneInfo]::ConvertTimeBySystemTimeZoneId([DateTime]::UtcNow, 'Korea Standard Time').ToString('yyyy-MM-ddTHH:mm:ss+09:00')"
```

자동 파이프라인이 생성하는 큐레이션 글은 원본 출처 목록을 담는 `source` 블록을 추가로 포함한다(정확한 스키마는 파이프라인 구축 시 정의).

## 운영 원칙 — 수정 전에 반드시 읽을 것

이 원칙들은 프로젝트의 지속 가능성을 지키기 위해 존재한다. 권장 사항이 아니라 제약이며, 미적 선호나 "이렇게 하면 더 좋을 텐데" 같은 충동보다 우선한다.

### 1. 테마는 고정. 커스터마이징 금지.

PaperMod 테마의 폰트·색상·여백·레이아웃 등 시각적 요소를 일체 수정하지 않는다. 테마는 배포된 상태 그대로 사용한다. 테마 업데이트가 필요하면 로컬 패치보다 깔끔한 upstream pull을 우선한다.

운영자가 명시적으로 시각적 변경을 요청하는 경우에 한해, `assets/css/extended/` 등 PaperMod의 공식 커스터마이징 지점에 가능한 한 작은 override로 구현한다. 테마를 fork하지 않는다.

### 2. 카테고리는 2개를 유지한다.

새 카테고리 추가를 절대 제안하지 않는다. 새 콘텐츠가 `curation`과 `note` 어디에도 맞지 않아 보이면, 답은 새 카테고리가 아니라 새 태그다.

### 3. 한 번에 한 단계만.

기능을 만들 때(일일 파이프라인, 커스텀 shortcode, 새로운 자동화 등) 한 부분을 구현·검증한 뒤에 다음으로 넘어간다. "하는 김에 같이" 식의 무관한 변경을 같은 커밋에 묶지 않는다.

### 4. 가상의 미래를 위해 미리 만들지 않는다.

"나중에 쓸지도 모르는" 빈 저장소·플레이스홀더 디렉터리·스텁 파일을 만들지 않는다. 지금 만들어야 할 구체적인 이유가 있을 때만 만든다.

### 5. 가벼운 도구가 무거운 도구보다 우선이다.

오늘의 문제를 해결하는 더 단순한 도구를, 미래에 가치를 발휘할지 모를 더 강력한 도구보다 우선한다. 예시:
- 큐레이션 파이프라인은 풀 프레임워크 대신 Python stdlib 위주.
- 데이터베이스 대신 마크다운 front matter.
- 별도 CI/CD 서비스 대신 GitHub Actions.

### 6. 커밋 컨벤션 (Conventional Commits + 한국어 description)

모든 커밋 메시지는 다음 형식을 따른다:

`<type>: <한국어 description>`

#### 허용되는 type (영어, 표준 키워드)

| type | 용도 |
|---|---|
| `post` | 새 글 추가 또는 글 내용 수정 |
| `feat` | 사이트 기능 추가 (검색, 댓글, 새 페이지 등) |
| `fix` | 버그 수정 |
| `style` | 시각·CSS 변경 (PaperMod override 등) |
| `chore` | 잡일 (의존성 업데이트, 설정 변경 등) |
| `docs` | 문서 변경 (README, CLAUDE.md 등) |
| `ci` | GitHub Actions 워크플로우 변경 |
| `build` | hugo.toml 등 빌드 설정 변경 |

#### description 규칙 (한국어)

- 한국어로 자연스럽게 작성한다.
- 명령형/현재형 어미를 권장한다: "추가", "수정", "갱신" 등 명사형 종결도 OK.
- 끝에 마침표를 붙이지 않는다.
- 제목 줄은 60자 이내를 목표로 한다 (한국어는 영어보다 정보 밀도가 높아 50자보다 약간 여유).

#### 본문(body)

추가 설명이 필요한 경우, 빈 줄 하나를 띄우고 본문을 작성한다. 본문은 한국어로 자유롭게 작성한다. 본문에는 "왜 이렇게 했는가"를 적는다.

#### scope

지금은 scope를 사용하지 않는다. 카테고리가 두 개뿐이라 description으로 충분하다. 향후 필요해지면 도입한다.

#### 예시

```
post: "ad meliora"에 대한 첫 노트 추가
feat: 사이드바에 태그 클라우드 추가
fix: hello 글의 깨진 링크 수정
style: 코드 블록 패딩 조정
chore: PaperMod submodule 업데이트
docs: CLAUDE.md 운영 원칙 갱신
ci: Hugo 버전을 0.141.0으로 갱신
build: 모든 섹션에 RSS 활성화
```

## 이 저장소에서 Claude Code 사용 시

### 기본 동작

- 모든 세션 시작 시 이 파일을 읽는다.
- 기능 추가 요청이 들어오면, 위의 운영 원칙과 충돌하는지 먼저 확인한다. 충돌이 있으면 구현 전에 그 충돌을 먼저 제기한다.
- 결정(테마, 구조, 의존성 등)에 확신이 없으면 추측하지 말고 묻는다. 운영자는 명시적인 트레이드오프 제시를 선호한다: "옵션 A는 X를 얻고 Y를 잃는다, 옵션 B는 그 반대다."

### 능동적으로 짚어줘야 할 신호

- 카테고리 목록을 늘리려는 모든 제안.
- 최소한의 CSS override를 넘어서 테마를 커스터마이징하려는 모든 제안.
- 현재 기능으로 정당화되지 않은 새 의존성을 추가하려는 모든 제안.
- "미래의 X를 위해 미리 준비하자" 패턴 중 X가 구체적으로 계획되지 않은 모든 경우.

### 하지 말아야 할 것

- 작동하는 코드를 스타일 선호 때문에 리팩토링하지 않는다.
- 명시적 요청 없이 새 빌드 도구·패키지 매니저·프레임워크를 도입하지 않는다.
- 대화에서 먼저 변경을 드러내지 않은 채 `hugo.toml`이나 `.github/workflows/deploy.yml`을 수정하지 않는다.

## 작업 모델·모드 convention

- **설계·계획 단계**: `/model opusplan`으로 들어가 Opus + plan mode에서 한다. 큰 구조 결정, 파이프라인 설계, 학습 로드맵 수립, 마이그레이션 계획 등은 이 모드에서 논의·확정한다.
- **실행 단계**: plan mode를 빠져나올 때 sonnet으로 전환한다. 확정된 plan을 따라 파일을 만들고 커밋하는 작업은 sonnet이 비용 대비 충분하다.
- **예외**: 실행 중 새로운 설계 결정이 튀어나오면 다시 `/model opusplan`으로 복귀한다.
- **Plan 기록 의무**: plan mode를 빠져나올 때, plan에서 논의·합의한 내용을 반드시 `plan_for_<주제>.md` 형태의 파일로 저장한다. 파일명은 주제를 명확히 드러내는 이름으로 한다 (예: `plan_for_curation.md`, `plan_for_curation_learning.md`). 이 파일은 이후 실행 단계에서 참조되며, 계획과 실제 구현의 차이를 추적하는 근거가 된다. 저장소 루트에 둔다.
